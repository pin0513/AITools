# Avoid AI Article - AI 生成文字準則

## 核心理念

> AI 生成的文章往往「安全、不冒犯、結構對稱、邏輯清楚」，看起來像「好文章」的模板。
> 但這種文章缺乏「密度」與「溫度」，讀者讀起來會覺得「空洞」「假」「沒靈魂」。

**本準則的目的**：讓 AI 協助撰寫的文章具有「人味」，讀起來像真人寫的。

---

## 第一部分：AI 味特徵識別

### 1. 典型的轉折與對比句型

AI 喜歡展現知識的「全面性」，所以非常愛用結構對稱的句子：

| 句型 | 問題 | 範例 |
|------|------|------|
| **「不是...而是...」** | 看起來有深度，其實空泛 | 「這不只是技術革命，而是認知的重塑」→ 什麼叫認知重塑？根本很空泛 |
| **「不僅...更...」** | 升級型句型，常接空洞結論 | 「不僅提升效率，更改變了工作型態」→ 如果「更」後面是「全面、深遠」就是 AI 味 |
| **「在某種程度上」「可以說是」** | 風險緩衝，沒有觀點 | AI 不確定就模糊化，實際上就是沒觀點，一直保留、不敢判斷 |
| **「然而，我們也必須意識到...」** | 準備講大道理的起手式 | AI 準備「轉折回中立立場」的標準套路 |
| **「關鍵在於...」** | 如果後面是誰都知道的廢話 | 「關鍵在於心態」→ 廢話！所以呢？ |
| **「這意味著...」** | 假裝推論完成 | 「這意味著我們必須重新定義成功」→ 誰說的？根據是什麼？ |

### 2. 罐頭結尾

AI 很怕文章沒結尾，所以結尾常出現這類「正能量且空泛」的呼籲：

| ❌ 絕對禁用 |
|------------|
| 「因此／總的來說／綜合以上，...」 |
| 「在未來的道路上，...」 |
| 「讓我們共同期待...」 |
| 「...需要我們每個人共同努力。」 |
| 「...關鍵在於如何在 A 與 B 之間取得平衡。」 |

**人類很少用「總的來說」做結尾**。人類寫到沒話講就停了，也不做結論。這種戛然而止反而很有人味。

### 3. 三段式或清單式的整齊結構

| AI 特徵 | 問題 |
|--------|------|
| 「首先／其次／最後」 | 每段長度幾乎一樣 |
| 「第一...第二...第三...」 | 結構異常整齊 |
| 「從 A 來看...從 B 來看...」 | 維度切分太工整 |

**人類的文章常常「不平均」**：有的段落特別長，有的特別短；寫著寫著突然岔開去講故事。

### 4. 抽象且宏大的修飾詞

| ❌ 絕對禁用 | 問題 |
|-----------|------|
| 「多維度」「全方位」 | 聽起來厲害但沒意義 |
| 「深遠影響」「賦能」 | 企業黑話 |
| 「範式轉換」「抓手」 | 中國式術語 |
| 「具有不可忽視的重要性」 | 廢話 |
| 「在當今快速發展的社會背景下」 | 萬能開場廢話 |
| 「為我們提供了一個全新的視角」 | 什麼視角？不說清楚就是空話 |

### 5. 溫和中立特徵

因為安全機制，AI 很少給出極端或偏激的評論：

| ❌ AI 的溫和 | ✅ 人的立場 |
|-------------|-----------|
| 「這是一個值得深思的問題」 | 「我認為這根本是假議題」 |
| 「每個人對此可能有不同的看法」 | 「但我的看法是...」 |
| 「這取決於多方面的因素」 | 「我認為最重要的是...」 |
| 「各有利弊」 | 明確說你偏向哪邊 |

## 第二部分：人味三問檢測

**寫完後問自己這三個問題**：

| 問題 | ✅ 有人味 | ❌ AI 味重 |
|------|---------|----------|
| **1. 有沒有具體細節？** | 有時間、地點、人物、數字 | 只有抽象描述 |
| **2. 有沒有作者的聲音？** | 有明確立場、情緒、個人經驗 | 溫和中立、各打五十大板 |
| **3. 敢不敢不完美？** | 有不確定性、有轉變、有承認錯誤 | 假裝什麼都懂、結構完美 |

## 第三部分：增加人味的技巧

> ⚠️ **重要區分：AI 協助 vs 人類創作**
>
> 以下技巧是給「人類作者」的建議。當 AI 協助寫作時，有不同的規則：
> - **AI 絕對不能編造場景、故事、個人經驗**
> - **所有具體細節必須來自用戶提供的素材**
> - **沒有素材就不要硬寫，改用客觀事實或技術描述**
>
> 詳見「第六部分：AI 協助寫作的邊界」

### 1. 加入個人的「經驗」與「故事」

AI 可以提供知識點，但案例、故事要你自己提供。

```
❌ AI 味：
「失敗是成功的母親，我們應從中學習。」

✅ 人味：
「我還記得那天創業失敗後，躲在便利商店吃冷掉的御飯糰，
眼淚掉在包裝袋上的聲音。那種鹹味，比任何教科書都讓我清醒。」
```

### 2. 展現明確的「立場」與「情緒」

一篇沒有觀點的文章，大家自己叫 AI 生成就好。你的觀點會讓別人開始喜歡你。

```
❌ AI 味：
「這件事各有利弊，我們應保持理性。」

✅ 人味：
「我認為一個政黨，不但阻擋了國防預算，還跑去敵國交流，
這根本是境外敵對勢力的在地協力者，或是說坦白點，就是間諜。」
```

### 3. 加上生活化的「譬喻」或「聯想」

AI 的邏輯是線性的，人類的思維常有跳躍。生活越豐富的人，文章能寫得越好。

```
✅ 範例：
「這段程式碼的邏輯，亂得像是我家客廳那堆永遠理不開的耳機線。」
```

### 4. 擁抱「不平均」

人類的文章好讀，是因為寫得「不平均」：

| 特徵 | 說明 |
|------|------|
| **有凸點** | 某段特別深入、特別精彩 |
| **有凹洞** | 某些地方故意略過不深究 |
| **有偏執** | 對某件事有強烈看法 |
| **有破綻** | 不完美但很真實 |

**AI 會平鋪直述地一個一個知識點丟出來，但人類寫著寫著，突然想到（靈感）就會岔開去講故事。**

### 5. 承認不確定性

AI 不敢承認自己不確定，但人類會：

```
✅ 「我到現在其實還沒完全想清楚這件事。」
✅ 「老實說，這件事一開始讓我很不舒服。」
✅ 「我以前很討厭這種說法，但後來發現我錯了。」
```

這種有時間軸、有過去和現在的比對和轉折，是很人類的特徵。

### 6. 把自己放進去

AI 沒有自己，所以會說「筆者認為」「我們」。但人類會說「我」。

```
❌ AI 味：「這個現象值得我們深思。」
✅ 人味：「老實說，這件事一開始讓我很不舒服。」
```

越把自己看得更重的，就越有人味。「筆者」這種詞會讓讀者覺得討厭——你人是到哪裡去了？

### 7. 不要強迫自己有結論

人類很容易寫到沒話講就停了：

```
✅ 「到現在，我還不確定這是不是好事。」
✅ （直接結束，不做總結）
```

AI 無法接受自己沒做到「有頭有尾」，但人類在表達上很願意擁抱不確定性。

## 第四部分：對照表速查

### AI 味 vs 人味

| 維度 | ❌ AI 味 | ✅ 人味 |
|------|---------|--------|
| 結構 | 整齊對稱、三段式 | 不平均、有長有短 |
| 立場 | 溫和中立、各打五十大板 | 明確站邊、敢有偏見 |
| 結尾 | 「總的來說」罐頭 | 戛然而止或留有懸念 |
| 修辭 | 「多維度」「賦能」 | 生活化譬喻、具體例子 |
| 情緒 | 理性、安全 | 有溫度、敢表達不滿 |
| 不確定性 | 假裝什麼都懂 | 承認「我也不確定」 |
| 時間軸 | 靜態描述 | 「以前我...後來我...」 |
| 自我 | 「筆者」「我們」 | 「我」 |

### 禁用句型速查

| 類別 | 禁用句型 |
|------|---------|
| 空洞轉折 | 不是...而是、不僅...更、在某種程度上 |
| 大道理起手式 | 然而我們也必須意識到、關鍵在於 |
| 假推論 | 這意味著、這說明了 |
| 罐頭結尾 | 總的來說、讓我們共同期待、需要共同努力 |
| 宏大修飾 | 多維度、深遠影響、賦能、範式轉換 |
| 萬能廢話 | 在當今快速發展的社會背景下、這是值得深思的問題 |

## 第六部分：AI 協助寫作的邊界

> **核心原則：AI 不是作者，是助手。助手不能代替主人說話。**

### AI 絕對不能做的事

| ❌ 禁止行為 | 說明 |
|------------|------|
| **編造場景** | 「去年某個週六下午，我在...」← 這種回憶 AI 沒有 |
| **虛構對話** | 「朋友跟我說...」← AI 沒有朋友 |
| **假裝經驗** | 「當我第一次使用這個工具時...」← AI 沒用過 |
| **捏造情緒** | 「那一刻我感到無比興奮」← AI 沒有那一刻 |
| **偽造細節** | 具體時間、地點、人名（除非用戶提供） |

### AI 可以做的事

| ✅ 允許行為 | 範例 |
|------------|------|
| **描述技術事實** | 「這個架構使用 Firebase + Next.js」 |
| **解釋原理** | 「這種設計模式的好處是...」 |
| **提供結構** | 幫助組織用戶提供的素材 |
| **潤飾文字** | 讓用戶的故事更流暢 |
| **標註留白** | 「[此處可加入您的經驗]」 |

### 當用戶沒有提供故事時

**選項 A：純技術/事實導向**
```
不要硬寫心路歷程，改用：
- 技術架構說明
- 功能特色介紹
- 問題與解決方案（從 code 推導，非編造）
```

**選項 B：標註留白**
```
在適合加入故事的地方標註：
「[建議：此處可加入您選擇這個技術的原因]」
「[建議：此處可分享您遇到的挫折]」
```

**選項 C：主動詢問**
```
「這個專案有沒有什麼特別的故事？
例如：為什麼選這個技術？中間有遇到什麼挫折嗎？」
```

### 範例對照

```
❌ AI 編造（絕對禁止）：
「去年某個週六下午，我在公司弄 Azure DevOps 弄到快瘋掉。
那個 YAML pipeline 怎麼設都設不對，我甚至懷疑人生。」

✅ 技術事實（可以）：
「傳統的 CI/CD 設定（如 Azure DevOps YAML）需要處理
複雜的配置檔，這是 Firebase Hosting 想要簡化的痛點之一。」

✅ 標註留白（更好）：
「[此處可加入您選擇 Firebase 而非傳統 CI/CD 的原因和經歷]」
```

## 第七部分：寫實寫作技巧

> 參考新聞寫作與文學創作的技法，讓文字更有畫面感和說服力。

### 新聞寫作原則

**1. 倒金字塔結構**
```
最重要的資訊放最前面，讓讀者第一句就知道重點。
不是「鋪陳、鋪陳、終於講重點」，而是「重點、補充、細節」。
```

**2. 五 W 一 H 具體化**
```
❌ 模糊：「這個專案花了一些時間完成」
✅ 具體：「三週內完成 MVP，包含 12 個 API endpoint」

具體的 Who/What/When/Where/Why/How 讓讀者信服。
```

**3. 引述與對話**
```
新聞會引用受訪者的話增加可信度。
但 AI 不能編造引述——除非用戶提供。

✅ 用戶提供：「團隊成員說：『這個架構太乾淨了』」
❌ AI 編造：「據說有人評論這個專案很棒」
```

### 文學敘事技巧

**1. 場景描寫（Show, Don't Tell）**
```
❌ Tell（告訴）：「這個 bug 很難修」
✅ Show（展示）：「console 噴了 47 行紅字，
   stack trace 指向一個我從沒看過的檔案」

但注意：AI 只能 show 用戶提供的素材，不能自己編。
```

**2. 節奏變化**
```
長句製造張力，短句製造衝擊。

「當 deploy 按鈕按下去的那一刻，
terminal 開始滾動，綠色的 log 一行一行爬上來，
我盯著螢幕，手指懸在鍵盤上方，
等待那個我已經看過一百次的錯誤訊息——
但它沒有出現。」

然後突然：

「成功了。」
```

**3. 懸念與轉折**
```
不要一開始就講結果，保留一點懸念：

「我本來以為這個專案會失敗。」（鉤子）
「...」（過程）
「結果它活了下來，而且活得比我預期的好。」（轉折）
```

**4. 隱喻與類比**
```
用讀者熟悉的事物解釋陌生概念：

「Serverless 就像叫 Uber，你不需要養一台車，
用的時候叫就好，用完就走。」
```

### 寫實主義三原則

| 原則 | 說明 | AI 的限制 |
|------|------|-----------|
| **觀察** | 從現實中提取細節 | AI 只能觀察 codebase，不能觀察「人」 |
| **選擇** | 選擇最有意義的細節 | AI 可以幫忙篩選，但素材來自用戶 |
| **呈現** | 用技巧讓細節活起來 | AI 可以潤飾，但不能無中生有 |

## 快速啟動

```
/avoid-ai-article

用途：
- 撰寫前：參考「增加人味的技巧」+ 確認有無用戶素材
- 撰寫中：對照「禁用句型速查」+ 不編造場景
- 撰寫後：進行「人味三問檢測」
- 審稿時：對照「AI 味 vs 人味」+ 檢查是否有編造

核心記憶：
1. ⚠️ AI 不能編造場景、故事、經驗
2. 有具體細節（來自用戶素材或技術事實）
3. 有作者聲音（立場、情緒、經驗——需用戶提供）
4. 敢不完美（不確定、轉變、承認錯誤）
5. 沒有素材就用技術事實或標註留白
```
